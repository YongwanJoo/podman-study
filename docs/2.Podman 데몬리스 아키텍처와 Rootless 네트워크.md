# Podman 데몬리스 아키텍처와 Rootless 네트워크

---

## Podman 데몬리스 아키텍처 실습

### 1. Podman 데몬리스 아키텍처 확인

```bash
ps aux | grep podman
```

위 명령어의 결과를 확인하면 `grep` 명령어 자체 외에 podman 관련 프로세스가 존재하지 않는다는 것을 알 수 있다.
이를 통해 Podman이 별도의 백그라운드 서버(데몬) 없이 동작한다는 것을 확인할 수 있다.

### 2. 간단한 alpine 컨테이너 실행 및 확인

```bash
podman run -it --name test-alpine alpine sh
```

위 명령어로 alpine 컨테이너를 실행하고 내부 쉘에 접속한 뒤, 새 터미널을 열어 `ps aux | grep podman`을 실행하면 `grep` 명령어 외에 별도의 podman 관련 데몬 프로세스가 없다는 것을 확인할 수 있다.

이 때, `/usr/bin/conmon`으로 시작하는 프로세스가 보이는데, 이는 **conmon(Container Monitor)** 으로 각 컨테이너의 상태 모니터링, 로그 수집, 종료 상태 감지를 담당한다. Docker의 중앙 집중식 `dockerd` 데몬과 달리, Podman은 컨테이너마다 개별 `conmon` 프로세스를 두는 방식으로 동작한다.

## Rootless 컨테이너

### 1. Rootless란?
Podman 컨테이너 내부에서는 해당 컨테이너를 띄운 사용자가 **root** 사용자처럼 활동한다. 하지만 호스트 리눅스 관점에서는 권한이 크게 제한된 **일반 사용자 프로세스**일 뿐이기 때문에 강력한 보안 격리가 가능하다.

### 2. subuid와 subgid
일반 사용자가 컨테이너 안에서 **root** 외에도 여러 사용자 권한을 가지려면 가상의 UID(User ID)가 여러 개 필요하다.
이를 위해 Linux는 각 사용자에게 **수만 개의 가상 UID 범위**를 할당하며, 해당 설정은 `/etc/subuid`와 `/etc/subgid`에 저장된다.

### 3. 내 계정의 subuid 확인해보기

```bash
# 내 계정이 할당받은 가상 UID 범위 확인
cat /etc/subuid
```

> `[사용자 이름]:100000:65536` 형태로 보인다면, 해당 사용자는 컨테이너 내부에서 100000번부터 시작하는 총 65536개의 가상 UID를 사용할 수 있다는 의미이다.

## Container Networking

### 1. Rootful (Bridge 모드)
- **root** 권한으로 컨테이너를 띄울 때는 리눅스 커널의 네트워크 스택을 직접 제어하여 높은 성능의 **브릿지(Bridge)** 와 **iptables** 규칙을 생성해 외부와 통신한다.
    - **Bridge**: Linux Kernel의 가상 스위치로, 컨테이너 간 통신 및 외부 네트워크 연결을 담당한다.
    - **iptables**: 리눅스 커널의 방화벽으로, 포트 포워딩과 NAT(Network Address Translation) 및 IP Masquerading을 담당한다.

### 2. Rootless (slirp4netns)
- 일반 사용자는 커널 네트워크를 제어할 권한이 없으므로 브릿지나 방화벽을 직접 조작할 수 없다.
    - 대신, 사용자 공간(User Space)에서 동작하는 **slirp4netns**를 통해 가상으로 패킷을 라우팅하여 인터넷과 통신한다.

> 최신 버전의 Podman에서는 **Netavark**를 기본 네트워크 스택으로 사용하여 이 성능 한계를 크게 극복하였다.

- **Slirp4netns**
    - **root** 권한이 없는 일반 사용자가 컨테이너 내부에서 인터넷을 사용할 수 있도록 해주는 프로그램
    - 애플리케이션 수준에서 패킷을 변환하여 호스트 네트워크로 전달하는 방식으로, 커널을 직접 거치지 않는다.
    - 패킷 변환 과정으로 인해 성능(속도)이 다소 느리고, 여러 컨테이너가 동시에 통신할 때 병목 현상이 발생할 수 있다.

- **Netavark**
    - Podman 4.0 이상에서 기본으로 사용하는 네트워크 관리 프로그램
    - **Aardvark** DNS 서버와 연동하여 컨테이너 이름만으로 서로 통신이 가능하다.
    - **Slirp4netns**에 비해 성능이 우수하고 더 다양한 기능을 제공한다.

### 3. 컨테이너 네트워크 실습

```bash
# 'my-net' 이라는 이름의 네트워크 생성
podman network create my-net

# 생성된 네트워크 목록 확인
podman network ls

# 특정 네트워크의 상세 정보 확인 (Netavark 방식 확인)
podman network inspect my-net
```

## Traefik과 ngrok으로 Rootless 컨테이너 개방하기

### 1. Traefik
컨테이너 특화 리버스 프록시로, MSA 환경이나 Docker/Podman과 같은 컨테이너 환경에서 널리 사용된다.

- **장점**
    - 설정 파일 없이 컨테이너의 라벨을 읽어와 동적으로 라우팅 규칙을 설정한다.
    - **Auto Discovery**: 컨테이너에 라벨만 붙여주면 Traefik이 자동으로 라우팅 규칙을 등록한다.
    - 설정 파일 수정이나 서버 재시작 없이 즉시 라우팅 변경이 반영된다.

### 2. ngrok
외부 네트워크에서 내부망으로 접속할 수 있게 해주는 터널링 프로그램이다.
복잡한 방화벽 설정이나 포트 포워딩 없이도 외부에서 내부 서버로 접속할 수 있게 해준다.

### 3. ngrok과 Traefik을 이용한 Rootless 컨테이너 개방 실습

#### 3-1. Podman 소켓 활성화
Traefik이 실행 중인 컨테이너들을 감시하려면 Podman의 API 통신 소켓을 활성화해야 한다.

```bash
# 일반 사용자 전용 Podman 소켓 활성화
systemctl --user enable --now podman.socket

# 소켓 파일 경로 확인
ls /run/user/$(id -u)/podman/podman.sock
```

#### 3-2. Traefik 컨테이너 실행
Traefik을 8080 포트로 띄우고, 호스트의 Podman 소켓을 컨테이너 안으로 마운트하여 실행 중인 컨테이너들의 상태를 감시하게 한다.

```bash
podman run -d \
  --name traefik-router \
  --network my-net \
  --security-opt label=disable \
  -p 8080:80 \
  -p 8081:8080 \
  -v /run/user/$(id -u)/podman/podman.sock:/var/run/docker.sock \
  docker.io/library/traefik:v2.10 \
  --api.insecure=true \
  --providers.docker=true \
  --providers.docker.exposedbydefault=false
```

- --api.insecure=true: Traefik의 관리자 페이지를 별도의 인증 없이 접속 가능하게 설정
- --providers.docker=true: Docker/Podman의 컨테이너 정보를 읽어와 라우팅 규칙을 설정
- --providers.docker.exposedbydefault=false: 기본적으로 컨테이너를 노출하지 않고, 라벨을 붙인 컨테이너만 노출
- -v /run/user/$(id -u)/podman/podman.sock:/var/run/docker.sock: 호스트의 Podman 소켓을 컨테이너로 마운트하여 상태를 감시
- --network my-net: my-net 네트워크에 연결
- -p 8080:80: 호스트의 8080 포트를 컨테이너의 80 포트로 연결
- -p 8081:8080: 호스트의 8081 포트를 컨테이너의 8080 포트로 연결 (8081은 Traefik 대시보드 접속용)
- docker.io/library/traefik:v2.10: Traefik 이미지
- --name traefik-router: Traefik 컨테이너 이름

> 이때, --security-opt label=disable을 추가하지 않으면 SELinux로 인해 Traefik이 Podman 소켓을 읽어오지 못하는 문제가 발생할 수 있다.

#### 3-3. 간단한 웹페이지를 띄우기 (index.html)

- 연습용 index.html 생성

```bash
echo "<h1>Hello Rootless World! Python Web Server + Traefik</h1>" > index.html
```

- Python 웹서버 컨테이너 실행

```bash
podman run -d \
  --name my-python-web \
  --network my-net \
  --label 'traefik.enable=true' \
  --label 'traefik.http.routers.mypython.rule=PathPrefix(`/`)' \
  --label 'traefik.http.services.mypython.loadbalancer.server.port=80' \
  -v $(pwd)/index.html:/workspace/index.html:Z \
  -w /workspace \
  docker.io/library/python:3.9-alpine \
  python -m http.server 80
```

> Traefik에 my-python-web 컨테이너의 라벨을 읽어와서 자동으로 라우팅 규칙을 설정하고 /로 접속하면 my-python-web 컨테이너로 연결되게 만든다.
> -v $(pwd)/index.html:/workspace/index.html:Z 옵션을 통해 파일 권한을 설정해줘야 접속이 가능하다.


#### 3-4. 호스트 내부에서 라우팅 연결 테스트
Traefik이 이미 라벨을 스캔하여 Python 컨테이너로 연결되는 라우팅 규칙을 설정했기 때문에, 호스트 내부에서 curl 명령어를 통해 Python 컨테이너로 접속이 가능하다.

```bash
curl http://localhost:8080
```

> curl 명령어를 실행하면 Python 컨테이너의 웹페이지가 출력된다.

#### 3-5. ngrok으로 외부에서 접속하기

- ngrok 가입 및 인증 토큰 발급
    - https://ngrok.com/에 접속하여 회원가입
    - 로그인 후 'Your Authtoken'에서 토큰 확인

- Rocky Linux에 ngrok 설치

```bash
# 1. 록키 리눅스(Linux 64비트)용 최신 ngrok 압축 파일 다운로드
curl -O https://bin.equinox.io/c/bNyj1mQVY4c/ngrok-v3-stable-linux-amd64.tgz

# 2. 압축 풀기
tar -xvzf ngrok-v3-stable-linux-amd64.tgz

# 3. 푼 실행파일(ngrok)을 시스템 전역에서 쓸 수 있게 /usr/local/bin 폴더로 이동!
sudo mv ngrok /usr/local/bin/

# 4. 설치 확인
ngrok --version
```

- ngrok 인증 토큰 등록

```bash
ngrok config add-authtoken [여기에_본인의_토큰을_붙여넣기]
```

> Forwarding 주소로 들어가 만든 페이지가 보인다면 통신 성공!
